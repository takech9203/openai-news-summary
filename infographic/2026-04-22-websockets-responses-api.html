🚧 Codex エージェントループの課題

Codex エージェントループは「計画 → 実行 → 観察 → 評価 → 修正」のサイクルを反復実行する。1 つのタスク完了に数十回の API 呼び出しが必要となり、HTTP ベースでは以下のオーバーヘッドが累積する。

1

TCP 接続の確立

新規接続のたびに 3-way ハンドシェイクが必要。数十ターンの往復で遅延が蓄積する。

2

TLS ハンドシェイク

HTTPS 通信のための暗号化ネゴシエーションが毎ターン発生。

3

HTTP ヘッダの反復送受信

リクエスト/レスポンスヘッダの転送が毎回必要で、帯域幅を浪費する。

4

プロンプトの再処理

毎回の API 呼び出しでモデルコンテキスト全体を再送信・再処理。長いコンテキストほどコストが支配的に。

5

ロードバランサの再ルーティング

リクエストごとに新たなルーティング判断が必要で、異なるサーバーに振り分けられる可能性がある。

🔌 WebSocket による永続接続

WebSocket プロトコルは HTTP のアップグレードメカニズムを通じて確立される全二重通信チャネルであり、エージェントループのような頻繁な往復通信に最適なプロトコルとなる。

🔗 接続の永続化

エージェントセッション全体を 1 つの WebSocket 接続で処理。TCP/TLS のハンドシェイクコストを初回のみに限定。

⚡ 低オーバーヘッドなフレーミング

HTTP ヘッダの反復送信に代わり、WebSocket フレームの最小限のヘッダ (2 - 14 バイト) でメッセージを送受信。

🔄 双方向ストリーミング

サーバーからのプッシュ型イベント配信と、クライアントからの非同期メッセージ送信を 1 つの接続で同時に実現。

🖧 サーバー親和性

同一 WebSocket 接続内のリクエストが同一バックエンドサーバーにルーティングされ、セッション状態を効率的に管理。

🗃 接続スコープキャッシュ (Connection-Scoped Caching)

WebSocket サポートにおける最大の技術革新。セッション存続期間中、サーバーサイドで以下の要素をキャッシュし、エージェントループの 2 回目以降のターンでは差分のみを処理対象とする。

🧠 KV キャッシュ

モデルの注意機構 (attention mechanism) の中間状態を GPU メモリ上に保持。新しいターンでは差分のみを計算。

📝 トークン化済みプロンプト

処理済みプロンプト部分のトークン列を保持し、再トークン化を回避。

🔧 ツール定義

セッション内で変化しないツール定義情報を初回のみ処理。

📈 キャッシュヒット率

後続ターンほどキャッシュヒット率が向上し、コンテキストが長いほど効果が顕著に。

🔁 ターンごとのキャッシュ動作

▶ ターン 1: 初回リクエスト

[System Prompt] + [User Message] + [Tool Definitions] を全て処理し、KV キャッシュを構築して接続に紐付けて保存。

全て新規処理: 2,000 トークン

▶ ターン 2: ツール実行結果を追加

既存コンテキストはキャッシュ済み。[Tool Call] + [Tool Output] の差分のみ処理。

差分処理: 200 トークン | キャッシュヒット: 2,000 トークン

▶ ターン 3: さらにツール呼び出し

コンテキストが増大しても差分のみ処理。キャッシュ済み KV 状態を参照。

差分処理: 300 トークン | キャッシュヒット: 2,200 トークン

▶ ターン N: 効率が累積的に向上

キャッシュされる KV キャッシュのサイズが大きいほど、再計算の回避による節約量が増大。

キャッシュヒット率: ターンごとに向上

💻 コードサンプル: HTTP vs WebSocket エージェントループ
HTTP ベースのエージェントループ (従来方式) Python 
from openai import OpenAI

client = OpenAI()

# HTTP ベースのエージェントループ (従来方式)
context = [
    {"role": "user", "content": "テストを修正してください。"}
]

while True:
    # 毎ターン、新しい HTTP リクエストを送信
    # コンテキスト全体を再送信する必要がある
    response = client.responses.create(
        model="codex",
        input=context,
        tools=[{"type": "shell", "shell": {"image": "python:3.12-slim"}}]
    )

    # ツール呼び出し結果をコンテキストに追加
    for item in response.output:
        context.append(item)

    if response.status == "completed":
        break

    # 次のターンでは、増大したコンテキスト全体が再送信される
    # → プロンプト処理コストがターンごとに増加
WebSocket ベースのエージェントループ (新方式) Python 
from openai import OpenAI

client = OpenAI()

# WebSocket ベースのエージェントループ (新方式)
async with client.responses.connect(
    model="codex",
    websocket=True,
    tools=[{"type": "shell", "shell": {"image": "python:3.12-slim"}}]
) as session:
    # 初回リクエスト
    await session.send({
        "type": "response.create",
        "input": "テストを修正してください。",
    })

    # 永続接続上でイベントを受信
    # 接続スコープキャッシュにより、差分のみ処理
    async for event in session.events():
        if event.type == "response.done":
            if event.response.status == "completed":
                print("タスク完了")
                break
            # 次のターンでは差分のみが処理される
            # KV キャッシュが保持されているため、
            # 既存コンテキストの再計算は不要
WebSocket 接続の完全なサンプル (イベントハンドリング付き) Python 
from openai import OpenAI

client = OpenAI()

# WebSocket を使用した Responses API 接続
async with client.responses.connect(
    model="codex",
    websocket=True,
    tools=[
        {
            "type": "shell",
            "shell": {
                "image": "python:3.12-slim"
            }
        }
    ]
) as session:
    # 初回のタスク送信
    response = await session.send({
        "type": "response.create",
        "input": "プロジェクトのテストを実行して結果を報告してください。",
    })

    # エージェントループ内の各ターンで同一接続を再利用
    # 接続スコープキャッシュにより、差分のみが処理される
    async for event in session.events():
        if event.type == "response.output_item.done":
            if event.item.type == "shell_call":
                print(f"[Tool] $ {event.item.command}")
                print(f"[Output] {event.item.output[:200]}")
            elif event.item.type == "message":
                print(f"[Agent] {event.item.content[0].text}")

        elif event.type == "response.done":
            # エージェントが追加のツール呼び出しを要求した場合
            if event.response.status == "requires_action":
                # 接続スコープキャッシュにより、
                # 既存のコンテキストは再処理不要
                await session.send({
                    "type": "response.create",
                    "input": event.response.required_action,
                })
            else:
                print("タスク完了")
                break

👨‍💻 開発者への影響

エージェントアプリケーションのパフォーマンス向上

ターンあたり 40 - 60% のレイテンシ削減により、特にコンテキストが大きいエージェントセッションで顕著な効果。数十ターンにわたるエージェントタスクではタスク全体の所要時間が大幅に短縮される。

SDK アップグレードの推奨

Python SDK v2.32.0 以降で WebSocket 接続をサポート。イベントハンドラ、自動再接続、メッセージエンキュー機能が提供される。非同期 (async/await) パターンを前提とした API 設計のため、既存の同期コードからの移行にはコードの書き換えが必要。

コスト効率の向上

接続スコープキャッシュによるプロンプト処理の削減は、計算リソースの節約にもつながる可能性がある。キャッシュされる KV キャッシュのサイズが大きいほど節約量も増大する。

運用上の考慮事項

WebSocket 接続にはタイムアウトや最大持続時間が設定される可能性がある。接続スコープキャッシュは GPU メモリを消費するため、同時接続数の増加に伴うリソース管理が必要。接続状態の監視 (接続数、レイテンシ、キャッシュヒット率) のための運用体制の整備が推奨される。

📌 通信方式の選択指針
📡

HTTP (REST)

単発の質問応答、1 ターンで完結するタスク。最もシンプルな実装。

📢

SSE

レスポンスのリアルタイムストリーミング。チャットボット等に適する。

推奨: エージェント用途
⚡

WebSocket

マルチターンのエージェントワークフロー、頻繁なツール呼び出し。Codex に最適。

OpenAI が公開した本エンジニアリング記事は、Responses API への WebSocket サポート導入がエージェントワークフローのパフォーマンスをどのように改善するかを技術的に詳述している。Codex エージェントループを具体例として、HTTP のリクエスト/レスポンスモデルに起因する累積的なオーバーヘッドが、WebSocket による永続接続と接続スコープキャッシュの組み合わせで効果的に解消されることを示した。


ターンあたり 40 - 60% のレイテンシ削減という成果は、数十回の API 往復が発生するエージェントワークフローにおいて特に大きなインパクトを持つ。本発表は、Python SDK v2.32.0 の WebSocket 機能強化、Codex エージェントループの設計、および同日発表の Codex Remote Connections と一連の技術スタックを構成しており、OpenAI のエージェントインフラストラクチャが「正しく動作する」段階から「高速に動作する」段階へと成熟しつつあることを示す重要なマイルストーンである。
