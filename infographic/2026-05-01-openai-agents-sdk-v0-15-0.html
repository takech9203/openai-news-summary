API 更新 v0.15.0

Agents SDK v0.15.0: ModelRefusalError によるモデル応答拒否の明示的ハンドリング

2026-05-01

モデルの応答拒否を暗黙的な空出力やリトライ地獄から解放し、明示的な例外とカスタムハンドラで開発者に制御を委ねる

概要

🛡️

ModelRefusalError の導入

モデルが応答を拒否した際に専用の例外クラス ModelRefusalError が発生するようになり、拒否と正常な空応答を明確に区別可能に

🔄

構造化出力の無限リトライ問題を解消

従来、拒否レスポンスが JSON パースに失敗し MaxTurnsExceeded まで SDK がリトライし続けていた問題を根本的に解決

⚙️

error_handlers で拒否をカスタム処理

error_handlers 辞書に model_refusal キーを指定し、拒否時のフォールバック値やカスタムロジックを安全に提供可能

💰

API コストの無駄遣いを防止

拒否を即座に検出してランループを中断するため、無駄なリトライによる API トークン消費を大幅に削減

Before / After 比較

❌ Before (v0.14.x)

  • テキスト出力エージェントで拒否が空文字列 "" として扱われる
  • 構造化出力エージェントで拒否が JSON パース失敗に
  • SDK が MaxTurnsExceeded までリトライし続ける
  • 開発者が拒否と正常な空応答を区別できない
  • 拒否が発生したこと自体が通知されない

✅ After (v0.15.0)

  • ModelRefusalError で拒否が明示的にサーフェス
  • 拒否理由が refusal プロパティで取得可能
  • ランループが即座に中断される
  • error_handlers でカスタム処理が可能
  • 構造化出力のフォールバック値を安全に提供

主な機能

🚨 ModelRefusalError

src/agents/exceptions.py に定義された専用例外クラス。拒否理由 (refusal メッセージ) を保持し、try/except でキャッチ可能

🎯 model_refusal ハンドラ

error_handlers 辞書の model_refusal キーにハンドラを登録。拒否時にフォールバック値を返すグレースフルデグラデーションを実現

🔍 拒否検出ロジック

turn_resolution.py に +48 行の検出ロジック追加。各ターンの応答で拒否をリアルタイムに検出しランループを即座に中断

✅ 構造化出力バリデーション

ハンドラが返すフォールバック値はエージェントの出力スキーマに一致する必要があり、SDK が自動でバリデーションを実施

コードサンプル

Python 基本的な拒否ハンドリング (テキスト出力) 
from agents import Agent, Runner

agent = Agent(
    name="assistant",
    instructions="ユーザーの質問に回答してください。",
)

# error_handlers で拒否をカスタム処理
result = Runner.run_sync(
    agent,
    "不適切なコンテンツのリクエスト",
    error_handlers={"model_refusal": lambda data: data.error.refusal},
)

# result.final_output にはモデルの拒否メッセージが格納される
print(result.final_output)
# 例: "申し訳ございませんが、そのリクエストにはお応えできません。"
Python 構造化出力エージェントでの拒否ハンドリング 
from pydantic import BaseModel
from agents import Agent, Runner


class AnalysisResult(BaseModel):
    summary: str
    confidence: float
    categories: list[str]


agent = Agent(
    name="analyzer",
    instructions="入力テキストを分析してください。",
    output_type=AnalysisResult,
)


# 構造化出力のフォールバック値を返すハンドラ
def handle_refusal(data):
    return AnalysisResult(
        summary=f"分析不可: {data.error.refusal}",
        confidence=0.0,
        categories=["refused"],
    )


result = Runner.run_sync(
    agent,
    "不適切なコンテンツ",
    error_handlers={"model_refusal": handle_refusal},
)

# result.final_output は AnalysisResult 型として検証済み
print(result.final_output.summary)

開発者への影響

🏗️ 構造化出力エージェントの安定性向上

従来、拒否がパース失敗として無限リトライを引き起こしていた問題が解消される。MaxTurnsExceeded エラーの削減により、API コストの無駄遣いが防止される

🔎 拒否理由の可視化

モデルがなぜ応答を拒否したかを refusal プロパティで直接確認できるようになり、デバッグとモニタリングが容易になる

🛡️ グレースフルデグラデーション

error_handlers を活用することで、拒否時にフォールバック値を返すグレースフルデグラデーションパターンを実装できる。本番環境でのユーザー体験の向上に寄与する

🔬 コンテンツモデレーションの改善

拒否を明示的にキャッチできるようになったことで、コンテンツモデレーションやセーフティフィルタリングのワークフローがより堅牢に構築できる

⚠️
破壊的変更の可能性: 従来 final_output == "" で拒否を判定していたコードは、ModelRefusalError が発生するよう動作が変わるため、アップグレード時にエラーハンドリングの見直しが必要です。テストコードも空文字列チェックから ModelRefusalError のキャッチへ更新してください。
