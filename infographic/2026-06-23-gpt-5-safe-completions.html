従来手法 vs Safe Completions

従来手法

  • 生成後フィルタリングによるブロック
  • 一律的な refusal (拒否応答)
  • 過剰拒否によるユーザー体験の低下
  • 文脈を考慮しない判断
  • レイテンシの増加 (二重処理)

Safe Completions

  • 生成過程での制約適用
  • 安全な代替出力の自動生成
  • 文脈に応じた柔軟な判断
  • ユーザー意図の尊重
  • 単一パスでの効率的処理

実装例: moderation パラメータの使用

safe_completions_example.py 
from openai import OpenAI

client = OpenAI()

# Safe Completions を有効化した API リクエスト
response = client.chat.completions.create(
    model="gpt-5",
    messages=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "ユーザーのリクエスト内容"}
    ],
    moderation={
        "mode": "safe_completions",
        "safety_level": "standard",       # "strict" | "standard" | "permissive"
        "cot_monitoring": True,            # CoT 安全性監視を有効化
        "deployment_profile": "enterprise" # リスクプロファイル指定
    }
)

print(response.choices[0].message.content)

# 安全性メタデータの確認
safety_info = response.choices[0].safety
print(f"Safety score: {safety_info.score}")
print(f"Constraints applied: {safety_info.constraints_applied}")
print(f"CoT interventions: {safety_info.cot_interventions}")

主要ユースケース

過剰拒否の低減

従来の二値的な refusal ではなく、安全な代替出力を生成することで、正当なリクエストが不必要にブロックされる問題を大幅に軽減。ユーザー満足度の向上と安全性の両立を実現

エンタープライズ安全保証

Deployment Simulation と連携したリスクプロファイルにより、企業固有のコンプライアンス要件に適合した安全性制約を自動的に適用。監査証跡の提供も可能

規制コンプライアンス

EU AI Act、日本の AI 事業者ガイドラインなど、各地域の規制要件に対応した安全性レベルの設定が可能。制約ルールのカスタマイズにより、業界固有の要件にも柔軟に対応
